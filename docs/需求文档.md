# 萤石摄像头老人夜间外出监护告警系统 — 需求文档

## 一、项目目标

在 Mac 本机运行一个 Python 程序，通过局域网 RTSP 协议接入家中萤石摄像头的实时视频流，在指定时段重点监护老人“杨孝治”是否出现在“电梯厅”摄像头画面中；一旦命中目标事件，立即触发电话告警，并同时保存截图和短视频作为证据。

本项目当前目标不是通用“陌生人告警”，而是“重点家庭成员定向监护告警”。一期优先保证可接入、可识别、可电话告警、可留证、可回溯，暂不追求多机部署、云端管理和复杂 UI。

---

## 二、核心场景

用户的核心诉求如下：

- 老人姓名：`杨孝治`
- 年龄：`77 周岁`
- 健康背景：患有阿尔兹海默，存在夜间或清早独自外出的风险
- 重点关注位置：`电梯厅`
- 重点关注时段：夜间与清晨时段，最终以配置为准
- 期望动作：一旦在指定时段发现杨孝治出现在电梯厅，立即电话告警

一期默认示例时段可采用：

- `22:00-07:00`

该时段只是默认示例，最终以配置文件中的实际值为准。

## 三、项目范围

### 3.1 一期范围

- 在一台 Mac 上运行
- 使用 Python 命令行程序启动
- 单次运行只监控 1 个摄像头
- 一期真机联调默认使用“电梯厅”摄像头
- 通过局域网 RTSP 拉流，不依赖萤石云
- 本地保存家庭成员照片、本地保存告警证据
- 在识别到“杨孝治出现在电梯厅指定时段”时触发电话告警
- 同时在终端输出明确日志，便于本地观察
- 保存截图和短视频证据
- 控制证据目录的磁盘占用上限

### 3.2 一期非目标

- 不做桌面图形界面
- 不做移动端 App
- 不做多摄像头并发监控调度
- 不做云端存储、账号体系、远程管理后台
- 不做通用陌生人入侵告警作为一期主目标
- 不做复杂的人体行为识别，如摔倒检测、徘徊检测
- 一期不考虑小米门锁接入

### 3.3 后续可扩展方向

- 其他家庭成员定向监护规则
- 通用陌生人告警
- 飞书、微信、短信等多通道通知
- 多摄像头轮巡或并发监控
- 小米门锁事件与摄像头告警的联动

## 四、前置条件

以下条件在开发前或上线前必须确认：

- 电梯厅摄像头已验证支持局域网 RTSP 拉流
- 已拿到目标摄像头可用的 RTSP 连接参数
- Mac 与摄像头处于同一局域网，网络稳定
- 电话告警所依赖的外部服务已开通且可用
- 已准备杨孝治的样本照片，每人至少 5-10 张，尽量包含不同角度、光照、距离和真实监控视角
- 若要降低误报，其他家庭成员也应准备样本照片，避免被误识别为杨孝治

如果目标摄像头无法稳定提供 RTSP 视频流，则一期无法上线。

## 五、功能需求

### 5.1 摄像头接入

- 摄像头信息写在配置文件中，至少包含名称、RTSP 地址
- 程序启动时通过命令行参数指定要监控的摄像头
- 单个进程一次只处理 1 个摄像头的视频流
- 启动后应在合理时间内完成拉流连接，若连接失败则报错退出
- 运行中若 RTSP 断流，程序应自动重连

补充规则：

- 摄像头名称必须唯一
- RTSP 地址允许带账号密码
- 自动重连默认每 10 秒尝试一次
- 连续重连失败时程序不中断，但需要持续输出日志

### 5.2 家人录入

- 家人照片放在指定目录下，每人一个子目录
- 程序启动时自动扫描该目录并建立已知人脸特征库
- 若某个人目录为空，启动时给出警告
- 若整个家人目录不存在或没有有效照片，程序应报错退出
- 杨孝治的人脸样本是必需项，缺失时启动失败

目录示例：

```text
known_faces/
  杨孝治/
    01.jpg
    02.jpg
  老婆/
    01.jpg
  儿子/
    01.jpg
```

### 5.3 识别逻辑

一期识别逻辑采用“三层判断”：

1. 先检测画面中是否出现人
2. 检测到人后，尝试做人脸检测与杨孝治识别
3. 若单帧结果不稳定，则使用连续多帧投票做最终判断

事件判断规则：

- 未检测到人：不告警
- 检测到人，但不在目标摄像头或不在目标时段：不触发电话告警
- 检测到人，且稳定识别为杨孝治：触发目标事件
- 检测到人，但稳定识别为其他家庭成员：不触发目标事件
- 检测到人，但未能稳定确认身份：一期默认不触发杨孝治外出告警，但要输出识别不确定日志，便于后续优化

识别策略要求：

- 杨孝治识别不能由单帧直接触发
- 应基于连续采样帧做投票或窗口聚合
- 若人脸相似度接近阈值，允许引入辅助画像做弱加权
- 辅助画像只能影响置信度，不得绕过人脸识别直接判定为杨孝治

推荐默认规则：

- 连续 3 帧中至少 2 帧识别为杨孝治时，视为命中
- 或在 2 秒时间窗内累计达到配置阈值时，视为命中

### 5.4 监护规则

一期仅要求实现一条核心监护规则：

- 目标人物：`杨孝治`
- 目标摄像头：`电梯厅`
- 目标时段：夜间与清晨，最终以配置为准
- 触发动作：电话告警 + 本地日志 + 证据保存

补充规则：

- 同一摄像头同一规则 5 分钟内不重复触发新的电话告警
- 冷却时间内允许继续写日志，但不重复拨打电话
- 非目标摄像头即使识别到杨孝治，也只记录日志，不触发该规则

### 5.5 告警方式

- 识别到目标事件后，优先触发电话告警
- 告警日志中需包含摄像头名称、事件时间、人物姓名、规则名称
- 同时在终端输出明确的本地告警日志，便于本机观察
- 同一摄像头 5 分钟内不重复触发新的电话告警

界面表现要求：

- 程序默认在命令行界面运行，不引入桌面 GUI
- 目标事件发生时，在终端输出明显的告警文本
- 告警文本至少展示以下信息：`[告警]`、摄像头名称、时间戳、人物姓名、规则说明、证据保存路径、电话告警结果

电话告警要求：

- 电话告警是一期核心能力，不再降级为后续项
- 若电话告警失败，必须记录失败原因
- 即便电话告警失败，截图和短视频仍然必须保存
- 电话告警实现需要保留失败重试或至少明确失败日志

### 5.6 证据保存

- 每次目标事件都要保存截图
- 每次目标事件都要保存短视频
- 文件名中包含时间戳和摄像头名称
- 所有证据保存到本地指定目录
- 证据目录占用磁盘空间必须受限，不能无限增长

视频规则明确如下：

- 保存告警前 5 秒与告警后 5 秒的视频片段
- 程序需维护内存中的环形视频缓冲区以保存告警前片段
- 告警触发后继续录制告警后片段，再合并为一个完整短视频
- 默认输出格式为 `mp4`

保留策略：

- 证据目录需配置最大占用空间
- 当证据目录达到上限后，程序自动删除最旧的历史证据文件
- 删除顺序按文件创建时间从旧到新执行，直到目录占用回到上限以下
- 清理策略应同时适用于截图和短视频
- 默认按单个 `evidence_dir` 总目录进行容量控制
- 一期默认最大占用空间可设为 20 GB，允许通过配置调整

### 5.7 启动检查与运行方式

- Mac 本机运行
- 命令行启动
- 支持通过参数指定摄像头名称
- 支持 `--check` 仅做配置、依赖、摄像头和电话告警能力检查
- 支持 Ctrl+C 优雅退出

启动示例：

```bash
python main.py --camera 电梯厅
python main.py --camera 电梯厅 --check
```

启动检查至少验证：

- 配置文件加载成功
- 指定摄像头存在
- RTSP 连通性通过
- 人脸样本目录存在且包含杨孝治
- 电话告警相关配置存在且可初始化

### 5.8 日志与异常处理

程序至少记录以下日志：

- 启动参数
- 当前选定摄像头
- 人脸库加载结果
- 杨孝治目标规则是否已启用
- RTSP 连接成功或失败
- 断流与重连情况
- 目标事件触发情况
- 电话告警成功或失败
- 证据文件保存路径
- 程序退出原因

异常处理要求：

- 配置文件格式错误时，启动失败并明确报错
- 指定摄像头不存在时，启动失败并明确报错
- 杨孝治样本目录为空时，启动失败并明确报错
- 电话告警关键配置缺失时，启动失败并明确报错
- RTSP 连接失败时，明确报错
- 运行中网络断开时，自动进入重连流程

### 5.9 小米门锁说明

- 一期完全不考虑小米门锁接入
- 当前需求、实现、测试和验收都不包含门锁能力
- 如需接入门锁，后续应单独立需求，不纳入当前阶段

## 六、配置文件结构

```yaml
# config.yaml
cameras:
  - name: "电梯厅"
    rtsp_url: "rtsp://admin:password@192.168.77.117:554/h265"
    monitor_rules:
      - rule_name: "杨孝治夜间外出监护"
        person_name: "杨孝治"
        alert_schedules:
          - start: "22:00"
            end: "07:00"
        actions:
          - "phone_call"
          - "terminal_log"
          - "save_evidence"

faces_dir: "./known_faces"
evidence_dir: "./evidence"

alert:
  cooldown_minutes: 5
  person_frames_threshold: 2
  person_window_seconds: 2

video:
  pre_seconds: 5
  post_seconds: 5
  output_format: "mp4"

storage:
  max_evidence_size_gb: 20

stream:
  reconnect_interval_seconds: 10

profiles:
  - name: "杨孝治"
    role: "重点监护对象"
    gender: "male"
    age: 77
    mobility: "slow"
    notes: "阿尔兹海默，夜间独自外出风险高"

phone_alert:
  provider: "aliyun_vms"
  enabled: true
  template_code: "TTS_xxx"
  called_numbers:
    - "13800000000"
```

配置要求：

- 配置加载失败时必须直接报错，不允许静默使用默认值
- `max_evidence_size_gb` 必须大于 0
- `monitor_rules` 至少包含 1 条针对杨孝治的规则
- `phone_alert.enabled = true` 时，关键配置必须完整

## 七、典型使用流程

1. 用户为杨孝治和其他家庭成员准备照片，放入 `known_faces/<姓名>/` 目录
2. 用户在 `config.yaml` 中填写电梯厅摄像头 RTSP 地址、监护时段和电话告警参数
3. 用户运行 `python main.py --camera 电梯厅`
4. 程序启动后加载人脸库并连接电梯厅摄像头
5. 程序持续分析视频流
6. 在目标时段内，如稳定识别到杨孝治出现在电梯厅：
   - 触发电话告警
   - 保存截图
   - 保存告警前后共 10 秒的短视频
   - 在终端输出本地告警日志
   - 在 5 分钟冷却期内不重复触发新的电话告警
7. 用户按 `Ctrl+C` 退出程序，程序完成资源释放并结束

## 八、验收标准

满足以下条件可视为当前一期需求完成：

### 8.1 基本可用

- 可通过命令行指定摄像头启动程序
- 可成功连接已验证可用的 RTSP 视频流
- 可成功加载杨孝治样本目录
- 可在终端或日志中看到运行状态

### 8.2 识别与告警

- 杨孝治在电梯厅的告警时段内出现时，触发电话告警
- 杨孝治在电梯厅的非告警时段出现时，不触发电话告警
- 其他家庭成员在电梯厅出现时，不触发杨孝治外出告警
- 杨孝治在非电梯厅摄像头出现时，不触发该规则
- 同一摄像头连续识别到杨孝治时，5 分钟内不重复触发新的电话告警
- 电话告警失败时，仍保留本地证据并输出明确失败日志

### 8.3 证据留存

- 每次目标事件都能生成截图
- 每次目标事件都能生成对应短视频
- 证据文件名包含时间戳和摄像头名称
- 证据文件可在本地目录中找到
- 当证据目录达到容量上限后，程序能够自动清理最旧证据
- 长时间运行后，证据目录总占用不超过配置上限

### 8.4 异常场景

- RTSP 断流后程序能够自动重连
- 电话告警依赖缺失或配置错误时，启动失败并明确报错
- Ctrl+C 退出时，程序能够正常释放资源

### 8.5 建议性能目标

- 单路 1080p 视频流下可持续运行
- 从目标事件满足触发条件到开始发起电话告警，目标不超过 3 秒
- RTSP 断流后恢复连接时间目标不超过 30 秒
