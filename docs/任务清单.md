### NotifyService 异步并发改造（2026-04-11 同日）
- 背景：首次回归时发现 `channel: ["dingtalk","ios_push"]` 请求耗时约 10 秒，触发 AppDaemon `Excessive time spent in callback` 警告。
- 约束固化：**所有第三方调用（HTTP / SDK / `self.call_service`）必须不阻塞 AppDaemon 事件循环**。
- 实现：
  - `on_notify_request` 改为 `async def`
  - 各通道通过 `asyncio.create_task(asyncio.to_thread(self._send_*, ...))` 派发到线程池
  - `asyncio.gather(..., return_exceptions=True)` 并发等待，单通道异常不影响其他通道
  - `_send_dingtalk` / `_send_ios_push` / `_send_phone` 保持同步实现（线程池隔离阻塞行为）
- 实测结果（对照组均为 2026-04-11 同一次部署的前后两次验证）：
  - `channel: "dingtalk"`：约 160 ms → 约 170 ms（单通道差异忽略不计）
  - `channel: ["dingtalk","ios_push"]`：**约 10 s → 约 1.4 s**，WARNING 消失
  - 两请求背靠背发出时时序几乎重合（12:23:01.897 / 12:23:01.917），互不阻塞
- 文档同步：`docs/notify-service-design.md` 新增 §10a 异步调度原则章节，明确未来新增通道的扩展约束。
