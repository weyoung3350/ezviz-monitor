# 自测清单

## 1. 用途

本清单用于约束 Claude Code 在交付 HA 一期实现前，至少完成与当前范围匹配的验证。

## 2. 基本要求

- 每项验证都要记录实际方法
- 若无法执行，要明确写出原因
- 自动化校验优先于口头说明
- 无法在当前仓库直接运行的 HA 验证，可用 trace、开发者工具、实际服务调用记录代替
- 验证结果需标明由哪个角色完成

## 3. 必做检查

### 3.1 实体与服务确认

至少确认：

- 门锁事件实体
- 门状态实体
- 摄像头快照方式
- Telegram 发送方式
- AppDaemon `NotifyService` 加载状态与监听方式

预期：

- 以上内容要么已确认可用，要么明确列为阻塞

**回填结果（2026-04-10，联调/验证工程师）：**

| 验证项 | 验证方式 | 结果 |
|--------|---------|------|
| 门锁事件实体 | HA API GET /api/states/ | ✅ 可用，有历史触发记录 |
| 门状态实体 | HA API GET /api/states/ | ✅ 可用，当前值"已上锁" |
| 摄像头快照方式 | 调用 camera.snapshot 服务并验证文件 | ✅ 可用，写入 /config/www/，HTTP 可访问 |
| Telegram 发送方式 | HA API GET /api/services/telegram_bot | ✅ 可用，telegram_bot.send_photo + file 参数 |
| NotifyService | AppDaemon 日志确认初始化 + 手动事件触发 | ✅ 可用，VMS 客户端初始化成功，事件监听正常 |

### 3.1b 统一通知服务

至少验证：

- `NotifyService` 已成功加载
- 能监听 `notify_service_request`
- Telegram 通道可正常发送
- 电话通道不可用时不会影响 Telegram

**回填结果（2026-04-10，方案负责人）：**

| 验证项 | 验证方式 | 结果 |
|--------|---------|------|
| NotifyService 加载 | AppDaemon addon 日志 | ✅ "NotifyService 已启动, VMS 客户端初始化成功" |
| 事件监听 | `POST /api/events/notify_service_request` | ✅ 参数正确解析（channel/request_id/source） |
| Telegram 文字通道 | `channel: telegram, message: "测试"` | ✅ 成功送达（Telegram 已解封） |
| 电话通道 | `channel: phone, phone_alert_name: "测试"` | ✅ 成功接到电话（CallId 已确认） |
| 双通道并行 | `channel: all` | ✅ Telegram + 电话同时成功 |
| 电话失败不影响 Telegram | phone_enabled=false 时 | ✅ Telegram 仍正常，phone.attempted=false |
| 结果事件 | 监听 notify_service_result | ✅ 包含双通道 attempted/success/error |

### 3.2 夜间门内开锁告警

至少验证：

- 夜间时段内，门内按钮开锁能命中自动化
- 时段外不触发
- 冷却期内不重复告警

**回填结果（2026-04-10，方案负责人）：**

| 验证项 | 验证方式 | 结果 |
|--------|---------|------|
| 夜间门内开锁触发 | 01:40 实际开门 + 事件监听 | ✅ H02 自动化正确触发，脚本启动 |
| 时段外不触发 | 白天测试（未在夜间时段） | ⏳ 待夜间端到端验证时同步确认 |
| 冷却生效 | 重置冷却时间戳后 3 分钟内再次开门 | ✅ 第一次触发，冷却期内不重复 |
| 三元组判断 | 实测事件属性：操作方式=9, 操作位置=1, 锁动作=2 | ✅ 完全匹配 |

备注：两次实测开门均在夜间时段（01:37 和 01:40），自动化正确触发。Telegram 首次被封导致脚本中断，加 continue_on_error 后第二次完整执行。

### 3.3 连续抓拍与失败兜底

至少验证：

- 能执行连续抓拍
- 至少成功 1 次时能发送图片通知
- 全部失败时仍有文字告警

**回填结果（2026-04-10，方案负责人）：**

| 验证项 | 验证方式 | 结果 |
|--------|---------|------|
| 连续抓拍执行 | 事件监听观察 5 次 delay 5s 循环 | ✅ 5 张 JPG 全部写入 /config/www/（每张 ~33KB） |
| 门状态持续观察 | 抓拍期间检查门状态捕获"已开锁"瞬态 | ✅ door_ever_opened=true |
| 抓拍成功发图 | HTTP 验证 /local/night_alert_*.jpg | ✅ 5 张全部可访问 |
| 全部失败兜底 | ⏳ 未模拟摄像头离线场景 | 待验证（需断开摄像头电源） |

备注：抓拍成功路径已完整验证。失败兜底路径需要物理断开摄像头才能触发，低优先级。

## 4. 推荐验证方式

可选方式包括：

- HA 自动化 trace
- HA 开发者工具手动触发服务
- 自动化运行日志
- Telegram 实际收到的消息
- YAML 配置检查
