# 老人夜间外出监护告警系统 — 需求文档 v2

> 基于 Home Assistant 中枢 + 多传感器融合架构，取代原独立 Python 程序方案。

## 0. Codex 评审修订说明

> **Codex 修改（2026-04-09）**
>
> 本次修订不改变“HA 自动化优先”的总体方向，主要补强以下几点：
>
> - 将“系统常驻运行”和“夜间强告警”边界写清，避免理解成全天候都要强提醒
> - 补充一期单次快照容易抓空的问题，改为短时间多次抓拍 + 失败兜底
> - 将一期冷却时间调整为“默认 5 分钟，可配置”，降低重复打扰风险
> - 将二期从“单传感器 + 老年男性判断”修正为“传感器时序 + 门锁/门状态佐证 + 结构化 AI 结果”的增强方案，更贴近独立电梯厅场景
> - 删除文档中的明文 API Key 要求，改为 `secrets.yaml`、环境变量或 AppDaemon 配置注入

## 一、背景

杨孝治，男，77 岁，患阿尔兹海默症，行动较慢。住四楼次卧，使用四楼次卫。家中已有 Home Assistant 2426 实体，包括人体存在传感器、智能门锁、萤石摄像头等设备。

原方案（独立 Python 程序 + RTSP 持续拉流 + 本地人脸识别）存在 CPU 100%、识别率低、每 6 分钟断流等问题，已放弃。

## 二、系统目标

【Codex 修改】系统常驻运行，但一期只对设定夜间时段内的疑似外出行为做强告警；白天行为默认不强提醒，以免产生大量无效通知。系统目标是尽早发现“夜间异常外出风险”，而不是对全天所有出入行为都发告警。

【Codex 修改】本系统关注的核心事件是：**杨孝治夜间独自外出**。若判断为有人陪同，则不属于本系统当前主告警目标。

【Codex 修改】除被动告警外，一期还包含一条主动预防能力：**从 23:00 起定期检查童锁状态，若未开启则发送 Telegram 提醒家人手动开启**，以降低夜间独自外出的发生概率。

## 三、分期规划

### 一期：规则触发告警（优先实现）

利用门锁事件 + 时段判断 + 摄像头快照，实现基础告警能力。

### 二期：AI 视觉增强

在一期基础上叠加通义千问 VL 云端视觉识别，通过传感器唤醒 + 路径上下文 + 按需快照分析，辅助判断是否为高风险外出场景，减少误报。

## 四、一期需求

### 4.1 监控范围

- 全天候监控，7×24 小时
- 告警仅在设定时段内触发：**23:00 ~ 07:30**（跨天时段）
- 告警时段可在 HA 中配置调整
- 【Codex 修改】在同一夜间时段内，系统还需定期执行“童锁守护”检查

### 4.2 触发条件

**单信号触发：** 门锁"门内开锁"事件

```
event: xiaomi_cn_1150511669_s20pro_lock_event_e_2_1020
条件:
  操作方式 = 9   (门内按钮开锁)
  锁动作 = 2     (开锁)
```

在告警时段内（23:00~07:30），满足上述条件即进入告警流程。

> 说明：门内开锁是从室内出门的最早、最确定信号（实测比门状态变化早 3 秒）。
> 当前阶段不区分"谁"开的门，通过快照可人工确认。
>
> **Codex 修改**：一期仍以“门内开锁”为主触发，不把门状态作为硬门槛，避免漏报；但建议在触发后额外观察 `sensor.xiaomi_cn_1150511669_s20pro_door_state_p_3_1021` 10 秒，若检测到开门/虚掩，可在通知文案中标记为“已确认开门”，提升人工判断效率。

### 4.3 告警动作

触发后执行以下动作：

1. **抓取电梯厅摄像头快照**
   - 实体：`camera.dian_ti_ting_mainstream`
   - 格式：JPEG
   - 已验证可用（ONVIF 集成，快照 API 正常）
   - 为降低”单次抓拍抓空”的概率，触发后连续抓拍多次，至少成功 1 次即发送图片通知
   - 默认参数：**5 次抓拍，每次间隔 5 秒**（共 20 秒），均为可配置项

2. **发送 Telegram 通知**
   - 接收人：当前用户（`chat_id` 通过 HA secrets 或运行配置注入）
   - 内容：文字描述 + 电梯厅快照图片
   - 文字示例：`⚠️ 夜间门内开锁告警\n时间：{{ now().strftime('%H:%M:%S') }}\n门锁状态：门内按钮开锁\n门状态确认：{{ 已确认开门 / 未确认门状态 }}`
   - **强制响铃**（`disable_notification: false`），不受 23:00~07:00 静默规则约束
   - 【Codex 修改】若全部抓拍尝试均失败，也必须先发纯文字告警，文案明确写出“快照抓取失败”，不能因为图片失败而吞掉告警

### 4.4 冷却机制

- 【Codex 修改】同一告警规则默认 **5 分钟内不重复触发**，并保留为可配置项
- 建议可配置范围：2~10 分钟
- 默认值采用 5 分钟，优先降低夜间重复打扰风险
- 超过冷却时间后如再次检测到门内开锁，重新触发告警

### 4.5 不触发告警的情况

- 告警时段外（07:30~23:00）的门内开锁
- 从室外开锁（指纹/密码/NFC/人脸等，操作位置=2）
- 上锁、关门、童锁操作等非开锁事件
- 关闭童锁事件（`锁动作=6`）**不触发告警**

### 4.6 夜间童锁守护

【Codex 修改】除“门内开锁告警”外，一期新增一条独立自动化，用于夜间持续检查童锁状态，并在未开启时提醒家人手动处理。

- 生效时段：默认 **23:00 ~ 07:30**
- 启动时点：每日 `23:00` 开始执行首轮检查
- 检查频率：默认 **每 10 分钟 1 次**，保留为可配置项
- 提醒冷却：默认 **30 分钟内不重复提醒**，保留为可配置项
- 判断条件：
  - 当前处于夜间时段
  - 童锁状态为未开启
- 执行动作：
  - 发送一次“童锁未开启，请手动处理”的 Telegram 提醒
  - 记录一次“童锁未开启提醒”日志
- 约束：
  - 童锁守护自动化与“门内开锁告警”自动化分开实现，避免职责混杂
  - 若童锁状态不可读，应明确记录为配置/集成缺失，不能静默跳过
  - 【Codex 修改】当前 `xiaomi_home` 未暴露童锁开启服务，因此一期不实现“自动补开”，而是降级为“检测 + 提醒”

### 4.7 实现方式

- HA 自动化 YAML（automation）
- 【Codex 修改】允许使用 HA `script` 配合自动化实现“等待门状态 / 连续抓拍 / 失败兜底 / 童锁提醒”，但一期仍不引入额外 Python 脚本或 AppDaemon
- 所有逻辑在 HA 内完成

## 七、外出路径分析

```
[四楼次卧] → [楼梯/电梯] → [三楼电梯厅] → [玄关] → [入户门] → 外出
                                  ↑              ↑           ↑
                            人体传感器      人体传感器    门锁事件
                            萤石摄像头                  操作方式=9, 锁动作=2
```

**起夜 vs 外出区分：**
- 起夜（四楼次卫）：不经过三楼，不触发三楼传感器
- 外出：经过三楼电梯厅 → 玄关 → 门锁开门

## 八、验收标准

### 一期

1. 告警时段内（23:00~07:30），从室内按钮开锁后，5 秒内收到首条 Telegram 告警；25 秒内（5 次抓拍周期）至少拿到 1 张有效快照或明确收到”快照抓取失败”的文字说明
2. 告警时段外，从室内开锁 → 不告警
3. 从室外开锁（指纹/密码/NFC）→ 不告警
4. 【Codex 修改】默认 5 分钟冷却期内重复开锁 → 不重复告警
5. 超过默认 5 分钟后再次开锁 → 重新告警
6. Telegram 通知强制响铃，不受静默规则影响
7. 【Codex 修改】夜间时段内，每 10 分钟执行一次童锁检查；若发现童锁未开启，则发送 Telegram 提醒手动处理
8. 【Codex 修改】同一童锁未开启提醒默认 30 分钟内不重复发送

### 二期

【Codex 修改】二期验收标准暂不在本版文档中细化，待一期稳定运行、二期需求冻结后再单独补充。

## 九、约束与边界

- 一期不区分"谁"开的门，依赖快照人工确认
- 一期不使用人脸识别，不拉视频流
- 【Codex 修改】童锁属于一期主动预防能力的一部分；系统在夜间时段内需要定期检查，并在未开启时提醒家人手动处理
- CW500 室外摄像头暂不接入（xiaomi_home 不支持视频流）
- 告警通道暂时只有 Telegram，电话告警后续独立需求
- 【Codex 修改】二期目前仅为预留增强项，不作为当前一期交付和验收阻塞项
- 【Codex 修改】若后续接入 AI，其结论属于辅助判断，不应脱离门锁、门状态或路径上下文单独下“已外出”结论
- 【Codex 修改】当前主目标只覆盖“独自外出”风险；若明确存在陪同人员，则记录观察日志，但不触发主告警
- 【Codex 修改】任何密钥、token、chat_id 等敏感信息应优先放在 HA secrets 或运行配置中，不以仓库明文为长期存储方式
