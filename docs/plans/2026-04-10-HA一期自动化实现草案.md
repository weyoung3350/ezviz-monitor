# HA 一期自动化实现草案

> 目标：基于 `docs/需求文档-v2-HA自动化.md`，落地一期能力，不展开二期 AI 细化。

## 1. 当前范围

一期只做以下两条能力：

1. 夜间门锁“门内开锁”告警

统一约束：

- 实现载体：Home Assistant `automation` + `script`
- 通知层统一接入 AppDaemon `NotifyService`
- 不引入与 HA 无关的独立外部后台
- Telegram 是当前已验证通道，电话通道通过 `NotifyService` 统一纳入
- 二期 AI 仅保留预留位，不进入本次实现

## 2. 设计原则

- 先保证规则稳定，再考虑界面美化或扩展能力
- 自动化与脚本分层，避免把所有逻辑塞进单条 automation
- 告警链路职责单一
- 所有敏感信息通过 `secrets.yaml` 或运行配置注入
- 对集成缺失、抓拍失败、服务调用失败必须有明确日志或通知

## 3. 建议结构

建议将一期拆成 1 条自动化、1 个脚本、1 组辅助实体。

### 3.1 自动化

1. `automation.night_door_unlock_alert`
   - 负责监听夜间门内开锁事件
   - 负责冷却判断
   - 负责调用告警脚本

### 3.2 脚本

1. `script.send_night_unlock_alert`
   - 连续抓拍
   - 生成统一通知请求参数
   - 记录结果

### 3.3 AppDaemon 服务

1. `NotifyService`
   - 监听 `notify_service_request`
   - 统一处理 Telegram / 电话通道
   - 统一处理静默规则、`force_sound` 和通道容错
   - 对外暴露统一事件接口，不让业务 YAML 直接绑定通知细节

### 3.4 辅助实体

建议使用以下辅助实体承载配置与运行状态：

- `input_boolean.night_unlock_alert_enabled`
  - 告警总开关
- `input_datetime.night_guard_start`
  - 夜间开始时间
- `input_datetime.night_guard_end`
  - 夜间结束时间
- `input_number.unlock_alert_cooldown_minutes`
  - 告警冷却分钟数，默认 `5`

如果不希望引入太多 UI 实体，也可以先把时间段和频率直接写在 YAML 中，后续再迁到 helper。

## 4. 实现链路

### 4.1 夜间门内开锁告警

触发源：

- `event.xiaomi_cn_1150511669_s20pro_lock_event_e_2_1020`

触发条件：

- 当前时间位于夜间时段
- 操作方式 = `9`
- 锁动作 = `2`
- 告警总开关开启
- 未处于冷却期

建议流程：

1. 自动化命中门锁事件
2. 校验夜间时段与冷却条件
3. 记录一次命中日志
4. 调用 `script.send_night_unlock_alert`
5. 更新“最近一次告警时间”

建议冷却实现方式：

- 优先用 `input_datetime.last_night_unlock_alert_at`
- 每次发送成功或进入发送流程时写入当前时间
- 自动化触发时比较当前时间与最近告警时间差值

这样比完全依赖 automation mode 更容易观察和调试。

### 4.2 告警脚本

`script.send_night_unlock_alert` 建议职责：

1. 立即发送首条文字告警
2. 在短时间内多次抓拍
3. 汇总图片路径、门状态确认结果和 `force_sound`
4. 发出 `notify_service_request`
5. 全部失败时保留纯文字告警路径，并补充抓拍失败日志

建议抓拍策略：

- 默认 `5` 次
- 每次间隔 `5` 秒
- 每次抓拍保存到 HA 可访问路径
- 抓拍成功即允许发送图片，不要求等全部轮次结束

建议通知文案至少包含：

- 告警标题
- 时间
- 门锁状态
- 门状态确认结果
- 快照抓取结果

门状态确认建议：

- 触发后等待最多 `10` 秒观察门状态实体
- 若出现开门/虚掩，则文案标记“已确认开门”
- 若未观测到明确变化，则标记“未确认门状态”

## 5. 配置建议

建议将以下内容沉淀为配置，而不是硬编码：

- 夜间开始/结束时间
- 抓拍次数
- 抓拍间隔秒数
- 告警冷却分钟数
- Telegram 接收对象
- 摄像头实体
- 门锁事件实体
- 门状态实体

统一通知服务建议配置外置到 AppDaemon `apps.yaml`：

- Telegram 目标实体或通道配置
- 电话通道凭据
- 静默开始/结束时间

如果第一版追求速度，可分两层：

- 第 1 层：YAML 常量先写死
- 第 2 层：逐步迁移为 helper 或 package 配置

## 6. 文件组织建议

若项目准备把 HA 配置一并管理，建议放入：

- `ha/packages/night_guard_automations.yaml`
- `ha/packages/night_guard_scripts.yaml`
- `ha/appdaemon/apps/notify_service.py`
- `ha/appdaemon/apps.yaml.example`

如果当前仓库还没有专门的 `ha/` 目录，也可以先新建 `docs/examples/` 存放示例 YAML，等接入实际 HA 仓库时再迁移。

## 7. 验证方案

实现后至少验证以下场景：

1. 夜间门内开锁，5 秒内收到首条 Telegram 告警
2. 连续抓拍至少拿到 1 张有效快照，或明确提示抓拍失败
3. 冷却期内重复门内开锁，不重复告警
4. 非夜间时段门内开锁，不告警

## 8. 当前阻塞项

实现前需要补齐或确认以下事实：

1. 快照文件写入路径和 Telegram 发送图片的可访问路径是什么
2. `NotifyService` 的 Telegram 最终走 `telegram_bot` 还是 `notify` 封装更稳
3. 电话通道在 AppDaemon 中的真实依赖和账号可用性

## 9. 建议实施顺序

1. 先确认门锁、门状态、摄像头的真实实体，以及 Telegram 发送方式
2. 再实现 `NotifyService` 与配置模板
3. 然后实现夜间门内开锁告警自动化和抓拍脚本
4. 最后做联调和验收记录

## 10. 结论

一期最合适的实现路径调整为：

- 用 HA 自动化处理触发、编排、冷却和抓拍
- 用 HA 脚本封装重复动作和通知参数整理
- 用 AppDaemon `NotifyService` 统一承接 Telegram / 电话通知

先把“门内开锁告警”稳定跑通，再决定是否进入二期增强。
