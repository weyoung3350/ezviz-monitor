# Claude Code 交付包说明

## 1. 用途

这份说明用于把当前项目的实现任务一次性交给 Claude Code。

目标是让 Claude Code：

- 先读清需求
- 按计划实现
- 按清单更新进度
- 完成自测
- 最后把结果交回给 codex 做 review 和验收

## 2. 建议交付顺序

把以下文件按顺序交给 Claude Code：

1. `docs/Claude-Code-启动提示词.md`
2. `需求文档.md`
3. `docs/plans/2026-04-08-萤石摄像头告警系统实现计划.md`
4. `docs/claude-code-执行说明.md`
5. `docs/自测清单.md`
6. `docs/任务清单.md`
7. `PROJECT_MEMORY.md`

`docs/验收测试方案.md` 不需要 Claude Code 自行发挥使用，但它可以知道后续会按这个方案被验收。

## 3. 每个文件的作用

### `docs/Claude-Code-启动提示词.md`

作用：

- 作为 Claude Code 的第一条启动消息
- 明确它应该先读哪些文档
- 明确当前一期目标
- 明确禁止事项
- 明确最终交付格式

### `需求文档.md`

作用：

- 定义最终要做成什么
- 定义一期范围和非目标
- 定义验收标准

### `docs/plans/2026-04-08-萤石摄像头告警系统实现计划.md`

作用：

- 定义任务拆分
- 定义模块边界
- 定义建议实现顺序
- 定义每个任务的最小测试

### `docs/claude-code-执行说明.md`

作用：

- 约束 Claude Code 的执行方式
- 规定不能擅自扩需求
- 规定每个任务完成后必须做什么

### `docs/自测清单.md`

作用：

- 约束 Claude Code 必须完成哪些测试
- 规定测试结果必须怎样汇报

### `docs/任务清单.md`

作用：

- 作为实现阶段的进度面板
- 作为 codex 后续 review 的主要入口

### `PROJECT_MEMORY.md`

作用：

- 提供跨会话上下文
- 说明当前阶段、已确认需求、建议架构和待验证事项

## 4. 发给 Claude Code 时的最简操作

最简方式：

1. 把 `docs/Claude-Code-启动提示词.md` 的内容直接发给 Claude Code
2. 确保以上文档都在同一项目目录中
3. 让 Claude Code 在项目目录内执行

如果需要附一句人工说明，可以只说：

```text
请严格按项目中的文档执行，先读启动提示词，再按任务清单推进并更新状态。完成后把修改、自测结果、风险和待 review 项回报。
```

## 5. Claude Code 完成后，codex 应重点检查什么

收到 Claude Code 的结果后，codex 优先检查：

1. `docs/任务清单.md` 是否逐项更新
2. 是否真的按当前一期目标实现成“命令行文字告警”
3. 自测结果是否包含真实命令和结果摘要
4. 是否偷偷跳过了证据磁盘上限控制
5. 是否把未完成内容如实列出

## 6. 不合格交付的典型表现

出现以下情况时，视为不合格，不进入正式验收：

- 没更新 `docs/任务清单.md`
- 只说“已完成”，没有测试命令和结果
- 把需求擅自扩成电话告警、GUI 或多摄像头
- 没实现磁盘占用上限
- 没说明阻塞和未完成项

## 7. 建议的协作闭环

建议按这个闭环执行：

1. codex 输出文档
2. Claude Code 按文档实现和自测
3. Claude Code 更新 `docs/任务清单.md`
4. codex 先基于任务清单做 review
5. codex 再按 `docs/验收测试方案.md` 做验收测试

这样后续返工也有统一入口，不容易丢上下文。
